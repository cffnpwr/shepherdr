# サービス管理

## 設定ファイル

サービス定義は`~/.config/shepherdr/config.toml`に置く（配置はdotfilesの責務）。
記述例は[config.example.toml](./config.example.toml)を参照。

`[[services]]`の各エントリは以下のフィールドを持つ。

| フィールド | 必須 | 型 | 内容 |
| --- | --- | --- | --- |
| `name` | ○ | 文字列 | サービス名。全エントリで一意。ログファイル名・UI表示に使う |
| `command` | ○ | 文字列配列 | 実行するargv。シェル評価しない |
| `login_shell` | - | 真偽値 | `true`でログインシェル経由で起動する。既定は`false` |
| `env` | - | テーブル | 追加の環境変数。appプロセスの環境に上乗せする |
| `cwd` | - | 文字列 | 作業ディレクトリ。未指定ならappプロセスのものを継承する |
| `enabled` | - | 真偽値 | `false`で定義を残したまま自動起動を止める。既定は`true` |

## spawn方式

[設計原則](./principles.md)のとおり、シェルを暗黙に挟まず`command`のargvをそのまま子プロセスとして実行する。

- `command[0]`が相対名の場合、appプロセスのPATH（LaunchServices経由の最小環境）で検索する（`posix_spawnp`相当）。
- appプロセスの環境にはログインシェルの設定（HomebrewのPATH等）が含まれない。ログイン環境が必要なサービスは`login_shell = true`を宣言する。

`login_shell = true`のサービスは、shepherdrが以下の形に組み立ててspawnする。

```text
/bin/zsh -l -c 'exec "$0" "$@"' <command[0]> <command[1]> ...
```

- argvはシェル文字列に埋め込まず引数として渡すため、シェル評価によるクォート事故が起きない。
- `command[0]`の解決はログインシェルのPATHに従う。
- zshは`exec`で`command`のプロセスに置換されるため、余計なプロセス層は残らない。
- ログインシェルの起動はサービスの（再）起動のたびに行われ、環境は毎回再構築される。
- `env`はspawn時にshepherdrが与えるため、ログインシェルの初期化スクリプトが同名の変数を設定する場合はそちらが勝つ。
- シェルはmacOS既定の`/bin/zsh`に固定する。

## 監視と再起動

- 子プロセスが終了したら、終了理由を問わず再起動する。再起動ポリシーの設定項目は設けない。
- 連続クラッシュ対策として再起動間隔に指数バックオフを適用し、一定時間の安定稼働でリセットする。具体値（初期間隔・上限・リセット閾値）は実装の既定値とする。
- トレイからの停止はそのサービスのdesired stateを「停止」に変え、再起動の対象から外す。
- desired stateは永続化しない。app再起動後は常に`config.toml`の`enabled`に従う（[設計原則](./principles.md)）。

## 設定の再読み込み

設定の反映はトレイメニューからの手動リロードで行い、差分を適用する。

- 追加されたサービスは起動する。
- 削除されたサービスは停止する。
- 定義が変わったサービスは再起動する。
- `enabled`が変わったサービスは起動または停止する。

## ログ

- 各サービスのstdout/stderrをappが捕捉し、`~/Library/Logs/shepherdr/<name>.log`へ書く。
- サイズ上限と世代数によるローテーションをapp側で持つ。具体値は実装の既定値とする。
- ログウィンドウ（[UI](./ui.md)）はこのファイルを読んで表示する。
